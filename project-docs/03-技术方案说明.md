# 技术方案说明

## 1. 技术路线

KeepDay 采用：

```text
Flutter + Riverpod + Drift(SQLite) + 本地通知 + JSON 文件导入导出
```

该方案适合本地优先、移动端优先、未来兼顾 Android / iOS 的产品形态。

## 2. 技术栈

| 能力 | 推荐技术 |
|---|---|
| UI 框架 | Flutter |
| 状态管理 | Riverpod |
| 路由 | go_router |
| 本地数据库 | Drift + SQLite |

## 4. 核心架构原则

### 4.1 UI 不计算业务规则

首页、月历、热力图不得自行判断宽限期、暂停期和连续天数。

统一由 Domain 层提供：

```text
StreakCalculator
GracePolicy
CalendarStatusResolver
HomeStatusResolver
```

### 4.2 写操作统一走 Use Case

所有会修改数据的操作都应通过 Application 层：

```text
CreateHabitUseCase
UpdateHabitUseCase
DeleteHabitUseCase
CheckInTodayUseCase
CancelCheckInUseCase
PauseHabitUseCase
ResumeHabitUseCase
RestoreBackupUseCase
```

Use Case 负责：

- 开启事务。
- 写数据库。
- 重算统计缓存。
- 更新通知计划。
- 返回 UI 状态。

### 4.3 统计可以缓存，但必须可重算

Habit 表保存：

- totalCheckInCount。
- bestStreak。
- currentStreak。

但缓存值只是加速展示，权威来源仍然是：

- CheckInRecord。
- HabitPausePeriod。
- Habit 当前 gracePerWeek。

如果缓存异常，应支持重新修复。

## 7. 本地通知方案

使用 `flutter_local_notifications`。

核心策略：

- App 启动后检查通知权限。
- 每次习惯提醒设置变化后重建该习惯通知。
- 暂停、删除习惯时取消该习惯通知。
- 恢复习惯时重新安排通知。
- JSON 恢复后取消全部旧计划，再按恢复数据重建。

Android 重点：

- Android 13+ 通知权限。
- 精确提醒能力。
- 开机后通知重建可后续增强。
- 厂商系统可能限制后台通知，产品文案需说明。

## 8. 备份恢复方案

备份文件为明文 JSON。

导出：

```text
查询未删除 Habit
→ 查询关联 CheckInRecord / NotificationLog / HabitPausePeriod
→ 生成 BackupPayload
→ 序列化 JSON
→ 调起系统保存或分享
```

恢复：

```text
读取 JSON
→ 解析 BackupPayload
→ 校验版本
→ 校验关联完整性
→ 用户确认
→ 数据库事务覆盖
→ 重算统计
→ 重建通知
```

## 9. 错误处理

写操作失败时：

- 保持原数据不变。
- 返回明确错误。
- UI 展示 Toast 或 Alert。

恢复失败时：

- 当前数据必须不被修改。
- 必须提示：“恢复失败，当前数据未被修改。”

## 10. 测试策略

优先覆盖 Domain 和 Application 层。

必须单测：

- 宽限期。
- 暂停期。
- 连续天数。
- 今日状态。
- JSON 校验。
- 恢复事务。

UI 测试可后置，但关键页面至少做 smoke test。
